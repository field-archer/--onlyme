更新概述：无人机协同任务下发与WS回传
========================

1) 任务下发（无人机/车队）
- POST `/api/uav/missions`（需 Bearer）
  - body：
    - mission_type：uav / fleet
    - waypoints：[{ longitude, latitude, altitude_m? }]
      - uav：每个点必须包含 altitude_m（米）
      - fleet：不需要 altitude_m
    - speed_level（可选）：low / medium / high
  - response data：{ mission_id: string }

2) WebSocket 回传（无人机协同）
- WS：`/ws/uav?token=<access_token>`
  - 结构：{ type, payload }
  - uav.telemetry payload：
    - longitude, latitude, battery, speed, altitude, roll, pitch, status(flying/hovering/landed)
  - uav.detection payload：
    - fire_probability, risk_level, flame_count, average_confidence, detected_target_count, longitude, latitude, fire_cause(human/lightning/farming/unknown)
  - fleet.arrived payload：{ arrived: true }

3) 返航
- 仍复用 POST `/api/uav/missions`：
  - 前端会生成“当前位置→Home”的 5 个航点下发（Home 为前端本地坐标，无需后端落库）

4) 车辆定位回传（可选）
- WS：`/ws/uav?token=<access_token>` 推送 `fleet.telemetry`
  - payload：{ longitude, latitude, ts? }
  - 说明：用于前端地图展示车辆当前位置；车辆任务派发仍复用 POST `/api/uav/missions`（mission_type=fleet，仅一个目标点）。

5) 演示用伪造检测服务（在后端实现时的验收口径）
- 前端不会在航线/遥测里「猜火点」；仅当收到 WS `uav.detection` 且 `flame_count>0`、坐标合法时，才会请求 POST `/api/fire-markers` 落库。若已收到过遥测且 `status` 为 `landed`（大小写不敏感）则不再落库，避免地面误触；尚无遥测时仍允许落库，以免「先发 detection、后补 telemetry」导致全程火点被挡掉。
- 「别的地方不能有火」由伪造服务控制发包：前端不按航点几何过滤检测坐标。
- 目标演示（与前端扫描航线生成一致，供伪造服务对齐）：
  - 扫描网格首行：航点序列为「左上 → 右上」（与 `generateScanWaypoints` 首行 row=0、先 left 后 right 一致）；仅在该段仿真轨迹附近发带火 `uav.detection`。
  - 全航线倒数第二段 → 倒数第一点：即航点列表中「倒数第二个航点 → 最后一个航点」的飞行段上发带火检测；其它航段不发 `flame_count>0`。
  - 仿真机已到达最后一个航点之后：不再推送带火检测（或仅推 `flame_count=0`）；避免前端在终点后仍落库。
- 联调顺序：仍建议先发 `uav.telemetry` 再发带火 `uav.detection`，便于 UI 状态一致；顺序颠倒时前端也会尝试落库（仅 `landed` 挡）。
