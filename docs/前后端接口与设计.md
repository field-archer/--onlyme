# 4.8（后端）接口与数据结构设计

## 4.8.1 模块接口（应用内分层）

后端按「路由 → 业务/校验 → 持久化」组织，主要目录职责如下。

| 模块路径 | 职责 |
|----------|------|
| `app/api/` | HTTP 路由：认证、火点、仪表盘、台账、火焰分析上传 |
| `app/core/` | 鉴权依赖、JWT、统一业务异常 `ApiError` |
| `app/models/` | SQLAlchemy ORM 模型（MySQL 表映射） |
| `app/schemas/` | Pydantic 请求/响应校验与序列化 |
| `app/services/` | 模型推理等业务（如 `/api/analyze`） |
| `config/config.py` + `config/config.yaml` | 应用配置（数据库连接、JWT、上传目录等） |
| `main.py` | FastAPI 入口、生命周期、全局异常处理、轻量库表同步（见下文） |

**模块间调用关系（简述）**：`app/api/*` 通过 `Depends(get_db)` 取得 `Session`，读写 `app/models/*`；认证路由调用 `app/core/security` 做密码哈希与 JWT；分析路由调用 `app/services/model_service`。

---

## 4.8.2 系统间接口（对外 HTTP API）

**Base URL（开发）**：`http://localhost:8000`  
**API 前缀**：由 `config.yaml` 的 `api_v1_str` 决定，默认 **`/api`**。  
**统一响应**：`{ "code", "message", "data" }`，成功 **`code === 20000`**。  
**鉴权**：除注册、登录、根路径、`POST /api/analyze` 外，下列接口需在 Header 携带 `Authorization: Bearer <token>`。

### 接口一览（与实现一致）

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| POST | `/api/auth/register` | 用户注册 | 否 |
| POST | `/api/auth/login` | 登录，返回 `access_token` | 否 |
| GET | `/api/auth/me` | 当前用户 | 是 |
| POST | `/api/analyze` | 图片/视频火焰分析（multipart） | 否 |
| GET | `/api/fire-markers` | 当前用户火点分页列表 | 是 |
| POST | `/api/fire-markers` | 创建火点 | 是 |
| PATCH | `/api/fire-markers/{id}` | 部分更新（含 `status/level/cause` 等；三者任一变化会写台账事件） | 是 |
| PATCH | `/api/fire-markers/{id}/status` | 仅更新状态；写台账事件 | 是 |
| DELETE | `/api/fire-markers/{id}` | 删除本人火点 | 是 |
| GET | `/api/fire-dashboard` | 仪表盘统计（全局聚合） | 是 |
| GET | `/api/fire-ledger` | 火情处置台账（事件表分页） | 是 |
| GET | `/api/geo/place-search` | 地名关键字搜索（后端代理图商，返回 POI 列表） | 是 |
| GET | `/api/geo/reverse-geocode` | 单点逆地理（区县级 `district`，后端代理图商） | 是 |

**OpenAPI**：服务启动后访问 `http://localhost:8000/api/openapi.json`（前缀以配置为准）。

## 4.8.3 消息字段（请求/响应约定）

### 4.8.3.1 统一 Envelope（所有 JSON 接口）

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 业务码；成功 `20000` |
| `message` | string | 人类可读说明 |
| `data` | object \| null | 成功为业务数据；失败常为 `null` |

### 4.8.3.2 认证相关（节选）

- **登录响应 `data`**：`access_token`、`expires_in`、`user` 等（详见 `docs/前端对接-用户与火点标记.md`）。
- **请求头**：`Authorization: Bearer <access_token>`。

### 4.8.3.3 火点 FireMarker（核心枚举）

| 字段 | 取值 | 前端展示（见产品文案） |
|------|------|------------------------|
| `status` | `pending` / `handling` / `extinguished` | 未处置 / 处置中 / 已扑灭 |
| `level` | `low` / `medium` / `high` | 低 / 中 / 高 |
| `cause` | `human` / `lightning` / `farming` / `unknown` | 人为用火 / 雷击火 / 农事用火 / 其他未知原因 |

**分页查询**：`GET /api/fire-markers` 与 `GET /api/fire-ledger` 均支持 `page`、`page_size`（当前实现单页上限 **200**，与前端大图列表拉取对齐）。

### 4.8.3.4 地区字段 `region`（区县级，方案 1：逆地理）

- **粒度**：以**区县级行政区划**为准（如「北京市密云区」）；**不按林场**拆分或统计。
- **写入**：创建火点 `POST /api/fire-markers` 及更新火点 `PATCH /api/fire-markers/{id}` 涉及经纬度时，由**后端**调用高德逆地理等 Web 服务（Key 仅存服务端）解析并写入 `fire_markers.region`；前端可传 `region: null`。
- **展示与统计**：台账、仪表盘 `region_bar` 等均按该字段聚合或展示；解析失败时主表可为空，事件表曾用占位「未知」（见 4.8.4.3）；**当前前端**在拿到 `marker_id` 或坐标且实现 `GET /api/geo/reverse-geocode` 时，会对台账与「各区县火情数量」做补全展示（仍以服务端持久化为准）。

### 4.8.3.5 地名搜索 `GET /api/geo/place-search`

- **Query**：`q`（关键词，必填或空时由后端定义行为）。
- **响应 `data`**：`{ id, name, address, location: [lng, lat] }[]`，与前端 `src/api/geo.ts` 类型一致；由后端代理图商关键字/输入提示类接口并规范化，**浏览器不直连**图商 REST 做 POI 检索。

### 4.8.3.6 逆地理单点 `GET /api/geo/reverse-geocode`

- **Query**：`longitude`、`latitude`（GCJ-02，必填）。
- **响应 `data`**：`{ district: string }`（或兼容 `region` 字段）；供前端在 `fire-ledger` 的 `region` 为占位、或 `fire-dashboard.region_bar` 不可用时，结合火点列表补全区县文案（见 `src/utils/regionEnrichment.ts`）。

## 4.8.4 数据库存储设计（MySQL）

数据库为 **MySQL 8.x**（推荐），连接串配置在 `config/config.yaml` 的 `database_url`。

### 4.8.4.1 表：`users`

存储注册用户；`password_hash` 为 bcrypt 哈希，不明文存密码。

### 4.8.4.2 表：`fire_markers`

火点主表：经纬度、标记时间、火焰处数、来源、备注，以及状态/等级/原因/地区、上报人冗余字段等。

**实现说明（与《后端须知》DDL 的差异，以代码为准）**：

- 归属字段：实现中使用 **`user_id`（VARCHAR(36)，外键关联 `users.id`）** 表示「该条火点归属用户」，功能上对应文档中的「标记人」语义。
- **`reporter_user_id`**：实现中为 **`VARCHAR(36)`**，与 `users.id`（UUID 字符串）一致，便于直接关联；若结题材料需写成 BIGINT，可在文档中说明「产品字段名」与「实现类型」的映射关系。
- 首次启动时：`main.py` 中 `Base.metadata.create_all()` 负责**新建表**；对已存在的旧库，另有**启动时补列**逻辑，避免仅改模型不迁移导致查询失败（生产环境建议逐步改为 Alembic 等正式迁移工具）。

### 4.8.4.3 表：`fire_marker_events`

火情台账事件表：`marker_id` 关联 `fire_markers.id`，记录某次处置相关展示字段及 `event_time`（与主表 `updated_at` 对齐写入）。

**业务规则（实现）**：

- `PATCH /api/fire-markers/{id}/status`：每次更新状态写入一条事件。
- `PATCH /api/fire-markers/{id}`：当 `status`、`level`、`cause` 任一相对原值发生变化时写入一条事件。
- 若主表 `region` 为空，事件表 `region` 历史上可用占位 **`未知`**；**推荐**写入事件时用与主表一致的区县级逆地理结果。`GET /api/fire-ledger` 建议在每条 item 中返回 **`marker_id`**（及可选 **`longitude`/`latitude`** 快照），便于前端与 `GET /api/geo/reverse-geocode` 联用补全展示。

# 4.8（前端）接口与数据结构设计

本节描述 **本仓库（Vue 3 + Vite）** 如何对接上述后端接口，对应结题材料中「模块接口 / 系统间接口 / 消息字段」的**前端视角**。

## 4.8.F.1 模块接口（前端应用内分层）

| 路径/文件 | 职责 |
|-----------|------|
| `src/api/client.ts` | 统一 `fetch`：`Authorization`、JSON、`code !== 20000` 抛 `ApiError` |
| `src/api/auth.ts` | 注册、登录、`/auth/me` |
| `src/api/fireMarkers.ts` | 火点 CRUD、`PATCH` 状态与综合更新 |
| `src/api/fireDashboard.ts` | `GET /api/fire-dashboard` |
| `src/api/fireLedger.ts` | `GET /api/fire-ledger` |
| `src/api/geo.ts` | `GET /api/geo/place-search`、`GET /api/geo/reverse-geocode`（鉴权） |
| `src/utils/regionEnrichment.ts` | 用火点列表与逆地理接口补全「各区县火情数量」与台账 `region` 展示 |
| `src/composables/useAuth.ts` | `localStorage` 持久化 `forestfire_token` / `forestfire_user` |
| `src/composables/useMap.ts` | 高德 JSAPI 加载、地图实例、火点覆盖物（需 `window.AMap`） |
| `src/views/HomeView.vue` | 地图大屏：布局、标记模式、拉取列表与看板 |
| `src/components/FireStatsCharts.vue` | ECharts 统计图（`echarts` npm 依赖，Vite 按需 `import()` 为独立 chunk，减轻首包） |
| `src/components/FireMarkerEditDialog.vue` | 点击火点后编辑 `status/level/cause` |
| `src/utils/fire.ts` | 状态/等级/时间等展示文案与格式化 |

**模块间数据流（简述）**：`HomeView` 在地图就绪后调用 `listFireMarkers` 绘制覆盖物，并缓存列表供 `regionEnrichment` 使用；标记模式下 `POST /api/fire-markers` 创建；侧栏地名搜索调用 `geoPlaceSearch`；点击标记打开 `FireMarkerEditDialog`，保存时 `PATCH /api/fire-markers/{id}`；看板与台账分别 `GET /api/fire-dashboard`、`GET /api/fire-ledger`，并对 `region_bar` / 台账 `region` 在必要时调用 `GET /api/geo/reverse-geocode` 补全区县展示。

## 4.8.F.2 系统间接口（前端 ↔ 后端 ↔ 第三方）

| 方向 | 协议/服务 | 说明 |
|------|-----------|------|
| 前端 → 后端 | HTTPS + JSON（及 `multipart` 分析接口） | `VITE_API_BASE_URL` 指向后端 `/api` 根；业务与地名搜索、逆地理相关能力均经此后端 |
| 前端 → 高德 | JSAPI Loader + WebGL 地图与瓦片 | `VITE_AMAP_KEY`、`VITE_AMAP_SECURITY_CODE`；初始化后需保证 `window.AMap` 可用（与覆盖物创建一致）。**不**经此后端发起高德 REST（POI、逆地理等由服务端完成） |
| 后端 → 高德（或等价） | HTTPS REST | Web 服务 Key 仅存服务端：逆地理写 `region`、供 `GET /api/geo/place-search` 与 `GET /api/geo/reverse-geocode` |

## 4.8.F.3 消息字段（前端发送与展示）

与后端 **4.8.3.3** 一致；前端在「标记火点」时发送示例：

```json
{
  "longitude": 116.397,
  "latitude": 39.909,
  "fire_count": 1,
  "source": "map_click",
  "status": "pending",
  "level": "low",
  "cause": "unknown"
}
```

编辑火点时 `PATCH /api/fire-markers/{id}` 可携带 `status`、`level`、`cause` 的任意组合，与后端「三者任一变化写台账事件」规则对齐。

## 4.8.F.4 前端状态与持久化（非数据库）

| 键名 | 存储 | 用途 |
|------|------|------|
| `forestfire_token` | `localStorage` | `Authorization: Bearer` |
| `forestfire_user` | `localStorage` | 顶部展示用户名等 |
| `forestfire_last_fire_count` | `sessionStorage` | 与首页检测流程衔接的火焰数量（可选） |

# 4.9（后端）系统部署说明

本节只写 **后端部署**；前端见下一节。

## 4.9.1 部署环境

- **操作系统**：Windows / Linux / macOS 均可。
- **Python**：3.9+（见 `docs/环境.md`）。
- **数据库**：MySQL 8.x；先执行 `scripts/init_mysql.sql` 建库。
- **依赖**：`pip install -r requirements.txt`（版本以仓库 `requirements.txt` 为准）。

## 4.9.2 配置与密钥

1. 编辑 `config/config.yaml`：`database_url`、`jwt_secret`、`jwt_expire_seconds`、`environment` 等。  
2. **生产环境**务必更换强随机 `jwt_secret`，并限制配置文件访问权限。  
3. 模型权重：按 `docs/环境.md` 放置检测用权重文件（文件名以该文档为准）。

## 4.9.3 启动命令

```bash
python main.py
```

或使用 uvicorn（等价）：

```bash
uvicorn main:app --host 0.0.0.0 --port 8000
```

## 4.9.4 验证与联调

- 健康检查：`GET http://localhost:8000/` 应返回 `code: 20000`。  
- 接口文档：`GET http://localhost:8000/api/openapi.json`。  
- 业务联调清单：注册 → 登录 → `GET /api/auth/me` → `POST/GET` 火点 → 仪表盘/台账。

## 4.9.5 运维注意

- **日志**：关注 MySQL 连接失败、JWT 校验失败、以及 `/api/analyze` 推理异常。  
- **依赖**：`passlib` 与 `bcrypt` 主版本需匹配（见 `requirements.txt` 注释），避免启动期无意义告警。  
- **数据迁移**：当前仓库含启动时「补列」兜底；正式上线建议引入 Alembic 等迁移方案并关闭自动 DDL。

# 4.9（前端）系统部署说明

## 4.9.F.1 部署环境

- **Node.js**：建议 **18+** 或 **20 LTS**（与 Vite 6 兼容）。
- **包管理**：`npm`（仓库含 `package-lock.json`）。

## 4.9.F.2 配置项（环境变量）

在仓库根目录创建 `.env` 或 `.env.local`（勿提交密钥到公开仓库）：

| 变量 | 说明 |
|------|------|
| `VITE_API_BASE_URL` | 后端 API 根，如 `http://localhost:8000/api`（末尾勿多余 `/`） |
| `VITE_AMAP_KEY` | 高德 Web 端 Key |
| `VITE_AMAP_SECURITY_CODE` | 高德安全密钥（JSAPI 2.0 必配） |

## 4.9.F.3 安装与本地运行

```bash
npm install
npm run dev
```

默认由 Vite 提示本地访问地址（常见为 `http://localhost:5173`）。

## 4.9.F.4 生产构建与预览

```bash
npm run build
npm run preview
```

构建产物在 `dist/`，可部署到任意静态资源服务器（Nginx、OSS+CDN 等）。**注意**：生产环境需将 `VITE_API_BASE_URL` 指向实际后端域名，并处理 **HTTPS** 与 **CORS**（后端需允许前端源）。

## 4.9.F.5 与后端联调检查清单

1. 后端已启动且 `GET /` 返回 `20000`。  
2. 前端 `VITE_API_BASE_URL` 与后端 `api_v1_str` 一致。  
3. 浏览器已登录（`localStorage` 有 `forestfire_token`），进入 `/map` 能 `GET /api/fire-markers` 拉取历史火点。  
4. 高德 Key/安全码有效，地图瓦片与覆盖物正常。  
5. `GET /api/fire-dashboard`、`GET /api/fire-ledger` 返回 `20000` 时，右侧统计与台账有数据；台账区域可独立滚动。

## 4.9.F.6 端侧 / 仿真与实机联调（简述）

- **端侧**：若大屏为国产浏览器或内网环境，需确认能访问高德域名（地图 Loader/瓦片）。ECharts 已随前端构建打包并按需加载，**不依赖**公网 ECharts CDN；内网部署时仅需同源加载构建产物即可。  
- **仿真**：前后端均在本机，按 4.9.F.5 清单验证。  
- **实机**：前端部署 HTTPS 域名，后端同样 HTTPS 或网关反代；核对 CORS 与 `Authorization` 是否被网关剥离。
